# 🌳 자료구조 구현 및 로우레벨 메모리 관리 고찰 (C++)

학부 전공 수업을 듣고 시스템 구현 역량을 기르기 위해, C++ 객체 지향 문법을 활용하여 바닥부터 직접 구현한 자료구조 저장소입니다. 

단순히 라이브러리를 가져다 쓰는 것에 그치지 않고, 포인터를 직접 제어하고 동적 메모리를 할당·해제하는 과정에서 발생할 수 있는 프로그램 오류와 예외 상황들을 깊게 고민하며 코드를 작성했습니다.

---

## 📂 주요 구현 프로젝트 목록

### 1. `BSTMap.cpp` (이진 탐색 트리 기반 Map)
- 라이브러리 없이 객체 지향 문법(`friend` 클래스 활용)으로 `Key(int) - Value(string)` 매핑 구조 설계
- 노드를 삭제할 때 자식 노드의 개수(0개, 1개, 2개)에 따른 3가지 예외 케이스 처리 및 소멸자 호출 시 메모리가 누수되지 않도록 후위 순회(Post-order) 방식의 해제(`clear`) 구현

### 2. `myfirst_AdjacencyGraph.cpp` (반복자 기반 인접 그래프)
- STL `list` 기반 그래프에서 발생하는 삭제 병목($O(N)$)을 해결하기 위해, Edge 클래스 내부에 자신의 위치를 나타내는 반복자(Iterator)를 직접 보관하도록 구조 개선
- 간선을 삭제할 때 처음부터 찾을 필요 없이 메모리 주소로 바로 접근하게 하여 진정한 $O(1)$ 연산 달성

## 🔍 핵심 고찰 및 문제 해결 과정

### 1. 학부 이론과 실제 구현의 괴리 해결 (삭제 연산 $O(N) \rightarrow O(1)$ 최적화)
- 교재나 이론으로 그래프를 배울 때는 간선 삭제나 인접성 검사가 상수 시간 또는 차수(Degree)만큼 걸린다고 단순하게 생각했습니다. 하지만 **실제 STL `list`를 활용해 직접 코드로 구현해 보니, 특정 간선을 지울 때마다 리스트를 처음부터 순회하며 찾아야 해서 $O(N)$의 병목이 발생**하는 것을 발견했습니다.
- 이론과 실제 성능의 차이를 해결하기 위해, `Edge` 클래스가 생성되는 시점에 **자신이 속한 리스트 내의 위치(Iterator)인 `pos_in_src`, `pos_in_dst`, `pos_in_edges`를 자기 내부에 직접 저장하도록 구조를 개선**해 보았습니다.
- 이렇게 하니 간선을 삭제할 때 별도의 탐색 과정 없이 메모리 주소로 곧바로 접근해 제거할 수 있게 되어, 대규모 데이터 처리 시에도 불필요한 지연이 생기지 않는 진정한 $O(1)$ 연산을 달성할 수 있었습니다.

### 2. 포인터 추적 및 동적 메모리 레이아웃 이해
- **BSTMap:** 자식이 2개인 노드를 삭제할 때, *"현재 노드가 오른쪽 자식일 때 부모를 타고 올라가며 처음으로 왼쪽 자식이 되는 시점의 부모가 Successor(후속 노드)가 된다"*는 예외적인 포인터 체인을 주석을 달아가며 직접 추적해 보았습니다.
- **Graph:** 이중 포인터 구조를 통해 Heap 영역에 포인터 배열과 연결 리스트가 어떻게 배치되는지 머릿속으로 메모리 레이아웃을 그려가며 구현하여 로우레벨 시스템 이해도를 높였습니다.

### 3. 세그멘테이션 폴트(Segmentation Fault) 및 취약점 방어
- 시스템 해킹 및 보안 취약점의 많은 원인은 잘못된 메모리 참조에서 발생합니다. 
- 본 프로젝트에서는 탐색 결과가 `nullptr`을 반환하는 예외 상황이나, 존재하지 않는 데이터를 참조·삭제하려 할 때 발생할 수 있는 **널 포인터 역참조(Null Pointer Dereference)** 에러를 막기 위해 방어적 조건문(`if (Node == nullptr)`)을 철저히 배치하여 소프트웨어의 안정성을 확보했습니다.

### 4. 이중 해제(Double Free) 및 댕글링 포인터 예외 처리
- 정점(Vertex)을 삭제할 때 연결된 간선(Edge)들이 연쇄적으로 지워지는 과정에서 발생할 수 있는 이중 해제 문제를 고민했습니다.
- 한쪽 정점이 먼저 사라지면서 간선 메모리를 해제할 때, 연결되어 있던 반대쪽 정점이 이미 파괴된 간선 메모리 주소를 참조하거나 다시 해제하려다 터지는 오류(널 포인터 역참조 등)를 막고자 했습니다.
- 상호 참조 관계의 링크를 안전하게 먼저 끊어주는 방어적 예외 처리 로직을 주석과 함께 구현해 보면서, 시스템 보안의 기본이 되는 메모리 안정성의 중요성을 체감할 수 있었습니다.
- 또한 `insert` 시 수행된 `new` 연산에 대응하여 `erase`와 소멸자에서 완벽하게 `delete`가 매칭되도록 설계했습니다.

### 5. 순회 중 컨테이너 수정으로 인한 이터레이터 무효화 방지
- 정점을 삭제할 때 연결된 간선들을 순회하며 제거하는 과정에서, `erase_edge`가 내부적으로 해당 정점의 `my_incident` 리스트를 직접 수정한다는 점을 발견했습니다.
- 원본 리스트를 순회하면서 동시에 그 리스트가 수정되면 이터레이터가 무효화되어 예상치 못한 오류(Undefined Behavior)가 발생할 수 있습니다.
- 이를 방지하기 위해 `erase_vertex` 내부에서 `my_incident`의 복사본을 먼저 만들어 순회하도록 구현했습니다. 순회 대상과 수정 대상을 분리함으로써 안전하게 모든 간선을 제거할 수 있도록 처리했습니다.
